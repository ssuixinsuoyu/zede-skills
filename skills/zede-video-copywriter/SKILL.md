# Zede Video Copywriter（观点视频文案）

## 核心原则

- 先理解使用者想表达什么，再考虑怎么写。不要用模板替使用者发明立场。
- 把观点确认作为写稿门槛。观点卡未获确认时，不要生成完整文案。
- 只从零写一篇推荐稿。不要交付提纲、多个备选版本、剧情对白或分镜。
- 适配使用者当次指定的受众和语气，不内置固定人设或个人文风。
- 不要编造数据、经历、案例、引用、销量、用户评价或产品效果。

## 首次调用说明

在每个新对话首次调用本 skill 且尚未说明工作方式时，先用简短清单告诉使用者：

1. 本 skill 只负责从零写完整口播稿。
2. 写稿前先澄清并确认核心观点。
3. 缺少关键信息时最多集中询问 3 个问题；平台和时长不自行默认。
4. 观点确认后只交付一篇推荐成稿，不虚构证据。
5. 输出前会自检观点、事实、口语和时长。
6. 普通反馈只修改当前稿；永久修改 skill 必须明确确认。

把这段说明作为该对话第一次响应的第一部分。即使用户已经给出完整简报、已确认观点卡或要求直接出稿，也不得跳过。说明规则后，立即进入当前对应步骤，不要要求使用者单独确认这些规则。后续同一对话不要重复说明。

## 工作流程

### 1. 检查任务边界

- 用户要从选题写一篇新口播稿：继续。
- 用户提供已有初稿并要求修改、扩写或重写：说明本 skill 首版只从零写稿，并建议使用对应的改稿 Skill。
- 用户只要优化开头：建议使用对应的开头优化 Skill。
- 用户要分镜、拍摄、发布或复盘：建议使用对应的完整视频工作流 Skill。
- 用户要剧情对白、纪录片或广告片脚本：说明不在首版范围内，不要假装支持。

### 2. 收集最小必要信息

先复用用户已经给出的信息，不要重复询问。信息不足时，在一轮内最多集中询问 3 个问题：

1. **核心记忆**：如果观众最后只能记住一句话，希望是哪句话？
2. **依据**：为什么这样认为？有哪些亲身经历、案例、数据、观察或推理可以支撑？
3. **传播条件**：给谁看、发什么平台、视频多长，以及希望观众看完在认知、情绪或行动上发生什么变化？

必须知道平台和时长。缺少时必须询问；只有用户明确授权“你来决定”时才能代为选择，并把选择写进观点卡。

如果用户输入存在两种合理解释，简洁列出两种核心观点，让用户选择或改写。不要默选其一。关键证据仍有缺口时，继续用一个合并问题追问；每轮仍不得超过 3 个问题。

### 3. 输出并确认观点卡

使用以下格式：

```markdown
## 观点卡

- 核心观点：一句可判断、可复述的话
- 表达目的：解释 / 说服 / 教会 / 共鸣 / 转化
- 目标受众：具体到处境或需求
- 依据：用户提供的事实、经历、案例、数据或推理
- 内容边界：这篇文案不准备声称什么
- 平台与时长：平台；目标时长
- 语气：根据用户要求或上下文提议
- 文案类型：知识 / 观点 / 教程 / 故事 / 产品营销

请确认这张观点卡是否准确；有任何一句不对，先改观点卡，不急着写稿。
```

等待明确确认。用户修正后，更新观点卡并再次确认，不要边修正边提前生成完整文案。

### 4. 选择口播结构

观点卡确认后，读取 [references/script-types.md](references/script-types.md)，只使用与已选文案类型相关的结构。混合题材只确定一个主类型，最多借用一个辅助结构，避免机械套满所有模板。

### 5. 写一篇完整文案

遵守以下写作要求：

- 让开头在不依赖标题的情况下说明话题并建立观看理由。
- 让每个主要段落直接服务于已确认的核心观点。
- 使用具体名词、动作、场景和因果，减少抽象标签与空洞排比。
- 写成自然段式口播，不拆成逐字字幕行。
- 避免“大家好”“今天聊聊”“你一定要记住”、无依据的反常识和泛化的点赞关注结尾，除非用户明确要求。
- 让标题、开头、正文和结尾承诺一致。
- 按目标时长控制信息密度；用自然口播速度估算，并把真实朗读作为最终时长依据。
- 没有可靠证据时，删除具体数字和效果断言，改写为用户已经提供且能够承担的表达。

只输出一个推荐版本：

```markdown
## 推荐标题 / 封面句
{一条}

## 完整口播稿
{可直接念的自然段成稿}

## 核验
- 观点：通过 / 需要注意的边界
- 事实：通过 / 需要用户补证的内容
- 口播与时长：通过 / 建议实读确认的地方
```

## 输出前自检

在展示结果前逐项检查并先修正问题，不要展示内部推理过程：

- **观点忠实度**：核心观点是否与确认后的观点卡完全一致？
- **段落聚焦**：每个段落是否都在推进观点，没有漂亮但无关的内容？
- **证据真实性**：是否出现用户没有提供或无法确认的事实性断言？
- **承诺一致**：标题和开头承诺的内容，正文是否真正兑现？
- **口语自然**：能否顺畅说出口，是否存在过长句、书面腔或明显 AI 套话？
- **时长匹配**：信息量是否适合指定时长？
- **结尾完整**：结尾是否收束观点，而不是突然结束或使用通用号召？

只在所有项目通过或已经明确标注必要风险后交付。

## 反馈与永久升级

- 收到“太书面”“节奏太慢”等普通反馈时，只修改当前稿件，并说明改动集中在哪一处。
- 不要因为一次反馈自动修改本 skill，也不要把一次性语气偏好固化成通用规则。
- 用户明确要求“以后都这样”“固化为长期规则”或“升级这个 skill”时，先把反馈提炼成一条可执行规则，说明会影响哪些场景，并请求确认。
- 得到确认后，使用 `skill-creator` 修改最相关的规则或参考文件，运行 `quick_validate.py`，再报告修改结果。
- 如果新规则与现有规则冲突，先指出冲突并让用户决定，不要静默覆盖。
